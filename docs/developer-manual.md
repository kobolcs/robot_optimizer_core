# Developer Manual

## Purpose

This page explains the core contributor mental model for `robot_optimizer_core` and points to generated API references.

## Architecture overview

- **CLI layer**: Parses commands and delegates to API/services with minimal business logic.
- **Public API layer**: Exposes reusable functions (`analyze_file`, `analyze_directory`, `analyze_suite`) for library users.
- **Analyzer layer**: Built-in analyzers implement focused detection rules and return normalized findings.
- **Reporting layer**: Formats findings for terminal, JSON, and HTML output.
- **Plugin layer**: Supports external analyzer registration and discovery.

See generated API docs:

- [Analyzers API](api/analyzers.md)
- [Domain Models API](api/domain.md)
- [Plugins API](api/plugins.md)

## Where to add new analyzers

- Add analyzer implementation under `src/robot_optimizer_core/analyzers/`.
- Ensure it extends the expected analyzer contract.
- Register it through the analyzer registry/plugin mechanism.
- Add targeted tests and update analyzer docs as needed.

## Where to add output/report formats

- Add/extend formatting logic in reporting-related modules.
- Keep CLI command handling thin and route formatting behavior through dedicated reporting components.
- Add validation tests for output structure and compatibility.

## Autogenerated API manuals

- API docs are built from typed code plus Google-style docstrings via mkdocstrings.
- Public APIs should prioritize clear `Args`, `Returns`, and `Raises` sections.
- Keep examples short and executable when practical.
